For each of the 14 API route files, apply these mechanical replacements:

1. **Add import** at top: `import { apiUnauthorized, apiError, apiBadRequest, apiValidationError } from "@/lib/api/errors";` (only import the helpers actually used in that file).

2. **Replace auth check pattern:**
   ```
   // BEFORE:
   if (!user) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
   // AFTER:
   if (!user) return apiUnauthorized();
   ```

3. **Replace database/server error pattern:**
   ```
   // BEFORE:
   return NextResponse.json({ error: error.message }, { status: 500 });
   // AFTER:
   return apiError(error.message, 500);
   ```

4. **Replace validation error pattern:**
   ```
   // BEFORE:
   if (err instanceof ZodError) {
     return NextResponse.json({ error: "Validation failed", issues: err.issues }, { status: 400 });
   }
   // AFTER:
   if (err instanceof ZodError) {
     return apiValidationError(err);
   }
   ```

5. **Replace generic bad request pattern:**
   ```
   // BEFORE:
   return NextResponse.json({ error: "Invalid request body" }, { status: 400 });
   // AFTER:
   return apiBadRequest("Invalid request body");
   ```

6. **Replace not found pattern (if any):**
   ```
   // BEFORE:
   return NextResponse.json({ error: "Not found" }, { status: 404 });
   // AFTER:
   return apiNotFound();
   ```

7. **Replace 501/other patterns:**
   ```
   // BEFORE:
   return NextResponse.json({ error: "Not implemented" }, { status: 501 });
   // AFTER:
   return apiError("Not implemented", 501);
   ```

After migrating, if a file no longer uses `NextResponse` for error responses but still uses it for success responses, keep the `NextResponse` import. Only remove it if the file no longer uses `NextResponse` at all (unlikely -- success responses still use it).

After all files are migrated, verify with grep that no inline error patterns remain.

Create `.github/workflows/ci.yml` with a CI pipeline that runs on push to main/master and on pull requests.

The pipeline should:
1. **Checkout** code
2. **Setup Node.js** 20.x with npm cache
3. **Install** dependencies (`npm ci`)
4. **Lint** (`npm run lint`)
5. **Type check** (`npx tsc --noEmit`)
6. **Build** (`npm run build`)
7. **Test** (`npm test`)

Required environment variables for build (set as empty strings since these are just for type/build checking, not runtime):
- NEXT_PUBLIC_SUPABASE_URL
- NEXT_PUBLIC_SUPABASE_ANON_KEY
- SUPABASE_SERVICE_ROLE_KEY
- ENCRYPTION_KEY
- GOOGLE_CLIENT_ID
- GOOGLE_CLIENT_SECRET

```yaml
name: CI

on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

jobs:
  ci:
    runs-on: ubuntu-latest

    env:
      NEXT_PUBLIC_SUPABASE_URL: "https://placeholder.supabase.co"
      NEXT_PUBLIC_SUPABASE_ANON_KEY: "placeholder-anon-key"
      SUPABASE_SERVICE_ROLE_KEY: "placeholder-service-key"
      ENCRYPTION_KEY: "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef"
      GOOGLE_CLIENT_ID: "placeholder-client-id"
      GOOGLE_CLIENT_SECRET: "placeholder-client-secret"

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npx tsc --noEmit

      - name: Build
        run: npm run build

      - name: Test
        run: npm test
```

Note: The ENCRYPTION_KEY must be a valid 64-char hex string (32 bytes) since the crypto module validates key length. The Supabase/Google values can be any non-empty strings since they won't be used for actual connections during CI.

IMPORTANT: The env validation module from Plan 02 runs on import, so these CI env vars must satisfy its checks. If the env module uses `createClient` at import time, the Supabase URL format may matter -- use a valid-looking URL.
